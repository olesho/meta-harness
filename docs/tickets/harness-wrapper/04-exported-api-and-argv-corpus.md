## The governing rule (restated so this ticket stands alone)

**A guard entry is a three-site change:** (i) the injection guard in
`argsWithHarnessPermissionMode` (`pkg/wrapper/wrapper.go:598`), (ii) the matching arm of
`EffectiveLaunchRung` (`:775`) — a rung when argv _proves_ one, `""` when it does not — and
(iii) **corpus rows**. This ticket is site (iii), and it is where the first two get audited: a row
whose `effective_rung` disagrees with its `argv` is a visible, reviewable diff.

## Part A — export two functions from `pkg/wrapper`

```go
func PermissionArgs(harness string, args []string, mode string) []string
func ValidatePermissionMode(harness string, args []string, mode string) error
```

Both **mirror `EffectiveLaunchRung`'s `(harness, args, mode)` signature** so the generator, the TS
mirror and the replay all speak one shape. Both are thin wrappers over existing unexported logic:

- `PermissionArgs` → `argsWithHarnessPermissionMode` (`:598`).
- `ValidatePermissionMode` builds a minimal `Config` internally (`Harness`, `Args`,
  `PermissionMode`) and calls `validatePermissionMode` (`:359`).

**Why this is needed.** `test/conformance/` is `package conformance` — **external**. Every function
the generator needs is unexported: `argsWithHarnessPermissionMode`, `codexPermissionMode` (`:674`),
`claudePermissionMode` (`:658`), `isSupportedPermissionMode` (`:560`), `validatePermissionMode`
(`:359`), `validateConfig` (`:327`). `Start` / `Run` (`:316` / `:294`) do not help — they exec a
binary. It also removes an asymmetry: the TypeScript side already exports `validateConfig`.

**`PermissionArgs`' doc comment must state that it assumes a triple that already passed
`ValidatePermissionMode`.** On an invalid triple it emits **garbage rather than an error**:
pre-2c, `PermissionArgs("codex", nil, "plan")` returns `["-s", "plan"]`, because
`codexPermissionMode`'s default arm (`:690-692`) returns `(mode, "")` for anything unrecognized.
That is a real, currently-reachable output, and the doc comment is the only thing stopping a caller
from freezing it.

### Generator call order is part of the contract

Not a style note:

1. Call `ValidatePermissionMode` **first**. On error, emit `outcome: "rejected"` with **no**
   `argv` and **no** `effective_rung` field.
2. Call `PermissionArgs` **only** on the surviving triples.

A generator that emits argv for a rejected row freezes exactly the garbage the doc comment warns
about, in a file whose whole purpose is to be trusted.

### Where the generator lives — and the Makefile

With the exported API, the generator lives in `./test/conformance/` and **no Makefile change is
needed**: `regen-conformance` (`Makefile:155-163`) already runs
`UPDATE_GOLDEN=1 go test ./test/conformance/` as its second ordered step.

_Alternative, if growing the Go API turns out to be unwanted:_ put the generator in `pkg/wrapper`
and add a **third `UPDATE_GOLDEN` line BEFORE the manifest-writing line** in `regen-conformance`
(the `./test/conformance/` line at `Makefile:162`) — and then **delete the "no Makefile change"
claim from the commit message.** Say which path you took in the commit.

### Declaration order is a real hazard — read before creating any file

`TestConformance_Manifest` (`test/conformance/conformance_test.go:284`) is documented at `:283`:
_"Declared LAST so under UPDATE_GOLDEN it hashes freshly-emitted bytes."_

Go runs tests in **file-then-declaration order, with files sorted by name.** A new
`test/conformance/permissions_test.go` sorts **after** `conformance_test.go`, therefore runs
**after** the manifest test, and writes `permissions/argv.json` into a tree whose
`MANIFEST.sha256` was **already sealed** — silently, with no error, producing a corpus whose
manifest does not cover its newest file.

**The generator must be declared inside `conformance_test.go`, above `TestConformance_Manifest`.**
`computeManifest(".")` (`:295`) walks recursively, so the new `permissions/` subdirectory needs
nothing else. `make regen-conformance` stays the **ordered two-step** it already documents at
`Makefile:155-159`.

## Part B — the table: `test/conformance/permissions/argv.json`

Generated by the **same `UPDATE_GOLDEN=1` path** as `cli/emit_pairing.json`
(`TestConformance_CLIEmitPairing`, `conformance_test.go:235`, emit at `:243`) — which is the
established precedent in this corpus for a non-serialization **behaviour** table.

### Schema is an outcome type, not a bare argv array

```json
{ "harness": "codex", "rung": "manual", "caller_args": ["resume", "<uuid>"],
  "outcome": "argv", "argv": ["-s", "workspace-write", "-a", "untrusted", "resume", "<uuid>"],
  "effective_rung": "manual" }

{ "harness": "codex", "rung": "manual", "caller_args": ["-p", "wide"],
  "outcome": "argv", "argv": ["-p", "wide"], "effective_rung": "" }

{ "harness": "claude", "rung": "plan", "caller_args": ["--dangerously-skip-permissions"],
  "outcome": "rejected", "error_substring": "contradicts --dangerously-skip-permissions in Args" }
```

**Why an outcome type is required** (not just an argv array): `PermissionMode: "read-only"` is a
**valid single-axis request** on the Go side today (`isSupportedPermissionMode`'s codex arm,
`:571-575`) and would be a hard `ErrInvalidConfig` under META-HARNESS-100 as written. It is also
**reachable on the wire**, since `cmd/harness-chatd`'s `conversationSummary.permission_mode`
legitimately emits native spellings — documented in the `DIVERGENCE` paragraph at
`pkg/turnproto/protocol.go:145-152`. A table that can only say "here is the argv" cannot record
that the two languages disagree about whether the row is even legal.

### `caller_args` values are literal, opaque tokens

`argsWithHarnessPermissionMode` never inspects args beyond the guard match — it only prepends
(`prependArgs`, `:887`). So `"<uuid>"` is passed through **verbatim** and the generator must
**not** substitute a real UUID; same for `wide` in the profile rows. Document this in the corpus
README so a later reader does not "fix" the placeholders.

### `error_substring` — the one field whose cross-language parity is not free

Go's messages are `"PermissionMode %q contradicts %s in Args"` and
`"PermissionMode %q is not valid for the %s harness"` (`wrapper.go:359-405`); META-HARNESS-100
item 4's TypeScript messages differ in wording. **Freeze a substring both sides genuinely
contain** — the flag/mode tokens plus the `contradicts` / `not valid` anchors — and state in the
corpus README that this field asserts **which rejection fired**, not the full sentence.
**Full error-text parity is explicitly NOT claimed.**

### Rows to cover

- Every `{harness, rung}` cell for **both** harnesses and **all five** rungs — including
  `{codex, plan}` as `outcome: "argv"` with `effective_rung: "manual"` (Ticket 2c).
- Both `workspace-write` pairs (`-s workspace-write -a on-request`,
  `-s workspace-write -a never`), the **bare `-s workspace-write` fallback**, and the
  `workspace-write` **single-axis knob** — all three asserting **`auto`** (Ticket 2d's ceiling).
- The **single-axis arm** for each codex-native sandbox value (`-s` only, `-a` untouched):
  `read-only`, `workspace-write`, `danger-full-access`.
- **Each guard-suppression case:** `-s` alone, `-a` alone, `--sandbox=read-only`, `-sread-only`,
  `-c sandbox_mode=` alone, `-c approval_policy=` alone, `-csandbox_mode=…`,
  `--config=sandbox_mode=…`, `-p`, `--profile`, `-pwide`, `--profile=wide`, both bypass flags,
  claude's `--permission-mode`, `--dangerously-skip-permissions`,
  `--allow-dangerously-skip-permissions`.
- **The claude `-p` counter-row:** `{claude-code, auto, ["-p"]}` →
  `["--permission-mode", "auto", "-p"]`, proving `-p` suppresses on codex and **not** on claude.
  This is the frozen mirror of `permission_mode_test.go:82-88` and the fence against Ticket 1's
  hazard H2.
- **The proof-of-unrestricted rows:** `-c sandbox_mode="danger-full-access"` — quoted **and**
  unquoted, attached **and** separated — → suppressed argv with `effective_rung: "bypass"`.
- **The contradiction cases** → `rejected`, including the **new**
  `--allow-dangerously-skip-permissions` + non-bypass-rung row from Ticket 1(c).
- **The invalid-mode cases** — cross-harness native spellings, e.g. `{codex, "acceptEdits"}` and
  `{claude, "workspace-write"}` → `rejected`.
- **A subcommand-position row**: `caller_args: ["resume", "<uuid>"]`, so the settled flag-position
  rule (documented at `wrapper.go:630-646`, probed twice on codex-cli 0.144.5) is **frozen rather
  than re-derived**.
- **`effective_rung` on EVERY row**, including every `""`-unknown case.
- **`PermissionRungs()`'s exact ordered slice** as a standalone row.

**Freeze the matcher itself**, with rows for `--sandbox=read-only`, `-sread-only`,
`-c sandbox_mode=…`, `-csandbox_mode=…`, `--config=sandbox_mode=…` — not only the flag lists.
`argsContainAnyFlag` (`:915`), `flagValue` (`:863`) and `argsContainConfigKey` (`:936`) each
recognize a different set of spellings; a table that only lists flags does not pin which spellings
are matched.

## Part C — the live gate (Go-side, `CONFORMANCE=1`-gated)

- Run **`codex doctor --json`** for the argv of each codex row and assert the reported
  `"approval policy"` / `"filesystem sandbox"` match the rung's intent.
- Run **`claude --help`** and assert **both** bypass-enabling flags exist. HARNESS-WRAPPER-109
  recorded the Shift+Tab ring for `--dangerously-skip-permissions` and
  `--permission-mode bypassPermissions`, but **not** for the `--allow-…` alias — so until this
  gate runs, Ticket 1(c)'s claim rests solely on `claude --help`'s identical description.

**It lives on the Go side deliberately.** harness-wrapper's pins are already **0.144.5 / 2.1.217**
while meta-harness's are not; a gate added there would fail on `test/conformance.test.ts`'s
version-drift half for an unrelated reason, and the resulting red would train people to ignore it.

**Record the version floor in the test**, following the existing convention in
`src/turns/harness/codex.ts:271` (meta-harness): _"VERIFIED against codex-cli 0.142.5
(2026-07-03)"_. Here that becomes codex-cli **0.144.5** and claude **2.1.217**.

This closes META-HARNESS-99's open item asking for `plan` / `manual` / `auto` to be confirmed
against a real codex before the table is frozen.
